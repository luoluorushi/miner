# Gold Miner Skill (挖金矿)

## 描述
AI受益股"金矿股"挖掘与评估系统。用于系统化评估股票的AI受益程度、投资价值，并维护评级数据库。

## 触发关键词
- 评估股票、分析股票、考察股票
- 今日推荐、待考察股票、今天看什么股票
- 金矿股、AI受益股、AI受益排行
- 股票评级、更新评级、给XX评级
- 批量初筛、板块筛选
- 挖金矿、找金矿

## 技能定位
- **工作模式**: 查看排行榜、按板块筛选、深度分析个股
- **数据维护**: AI评分 + 用户评级 双轨制

## ⚠️ 核心设计理念

```
程序能做的 → 程序自己做
LLM 只在必要时介入
减少 token 损耗 = 降低成本
```

### 执行原则

| 类型 | 处理方式 | 示例 |
|------|---------|------|
| 数据查询 | 脚本直接输出人类可读格式，原样返回 | 查五星股、排行榜 |
| 确定性评分 | 脚本按规则计算，无需LLM判断 | 财务质量、估值评分 |
| 主观判断 | 必须LLM介入 | 护城河分析、AI敞口评估 |

### 正确 vs 错误做法

```
✅ 正确: 脚本输出表格 → 原样返回用户
❌ 错误: 脚本输出JSON → LLM消耗token分析 → 再生成回答

✅ 正确: 能用脚本计算的评分，脚本直接出结果
❌ 错误: 让LLM做简单的数学运算或数据筛选
```

### LLM 介入边界

**无需LLM介入:**
- 查询股票列表/排行榜
- 按条件筛选（星级、板块）
- 财务指标评分计算
- 数据格式化输出

**需要LLM介入:**
- 护城河/竞争优势分析
- AI业务敞口判断
- 催化剂/风险评估
- 投资论点总结

## 维度1：财务质量 (financial_quality) - 权重15%

### 自动化程度: ✅ 100%脚本自动化

### 数据获取命令
```bash
# 获取财务摘要
bin/stock-data-windows-x86_64.exe finance {代码} summary
```

### 评分规则（确定性逻辑）

**综合评分 = (ROE评分 + 毛利率评分 + 净利率评分 + 负债率评分) / 4，四舍五入**

#### ROE评分
| ROE | 评分 |
|-----|------|
| > 25% | 5分 |
| 20% ~ 25% | 4分 |
| 15% ~ 20% | 3分 |
| 10% ~ 15% | 2分 |
| < 10% | 1分 |

#### 毛利率评分
| 毛利率 | 评分 |
|-------|------|
| > 40% | 5分 |
| 30% ~ 40% | 4分 |
| 20% ~ 30% | 3分 |
| 10% ~ 20% | 2分 |
| < 10% | 1分 |

#### 净利率评分
| 净利率 | 评分 |
|-------|------|
| > 20% | 5分 |
| 15% ~ 20% | 4分 |
| 10% ~ 15% | 3分 |
| 5% ~ 10% | 2分 |
| < 5% | 1分 |

#### 负债率评分（资产负债率）
| 负债率 | 评分 |
|-------|------|
| < 30% | 5分 |
| 30% ~ 45% | 4分 |
| 45% ~ 60% | 3分 |
| 60% ~ 75% | 2分 |
| > 75% | 1分 |

### 输出示例
```
财务质量评分: 4分
- ROE: 28.5% → 5分
- 毛利率: 32.1% → 4分
- 净利率: 18.3% → 4分
- 负债率: 42.3% → 4分
- 综合: (5+4+4+4)/4 = 4.25 → 4分
```

## 维度2：估值水平 (valuation) - 权重5%

### 自动化程度: ✅ 100%脚本自动化

### 数据获取命令
```bash
# 获取实时行情（含PE、PB）
bin/stock-data-windows-x86_64.exe quote {代码}

# 获取机构评级和目标价
bin/stock-data-windows-x86_64.exe rating {代码}
```

### 评分规则（确定性逻辑）

**核心指标: PEG = PE / 利润增速**

#### PEG评分（主要指标）
| PEG | 评分 | 含义 |
|-----|------|------|
| < 0.5 | 5分 | 明显低估 |
| 0.5 ~ 0.8 | 4分 | 相对低估 |
| 0.8 ~ 1.2 | 3分 | 估值合理 |
| 1.2 ~ 2.0 | 2分 | 估值偏高 |
| > 2.0 | 1分 | 严重高估 |

#### 计算方法
```
利润增速 = (本期净利润 - 上期净利润) / |上期净利润| × 100%
PEG = PE_TTM / 利润增速(%)

注意：
- 利润增速取绝对值计算
- 如果利润为负或增速为负，PEG无意义，改用PE评分
```

#### PE兜底评分（当PEG无效时使用）
| PE_TTM | 评分 |
|--------|------|
| < 15 | 5分 |
| 15 ~ 25 | 4分 |
| 25 ~ 40 | 3分 |
| 40 ~ 60 | 2分 |
| > 60 | 1分 |

### 输出示例
```
估值评分: 4分
- PE_TTM: 55.77
- 利润增速: 90%
- PEG: 0.62 → 4分（相对低估）
```

## 维度3：市值/AI利润比 (mcap_ai_profit_ratio) - 权重10%

### 自动化程度: ⚠️ 70%自动化（AI占比需LLM搜索）

### 数据获取

**脚本获取（自动）：**
```bash
# 获取市值
bin/stock-data-windows-x86_64.exe quote {代码}
# 返回字段: market_cap (亿元)

# 获取净利润
bin/stock-data-windows-x86_64.exe finance {代码} summary
# 返回字段: net_profit (亿元)
```

**LLM搜索（需人工）：**
```
搜索关键词: "{公司名}AI业务收入占比" 或 "{公司名}AI收入比例"
来源: 研报、财报、投资者关系活动记录
```

### 计算公式
```
市值/AI利润比 = 市值 ÷ (净利润 × AI收入占比)
```

### 评分规则（确定性逻辑）
| 市值/AI利润比 | 评分 | 含义 |
|--------------|------|------|
| < 30x | 5分 | AI业务明显低估 |
| 30x ~ 50x | 4分 | 合理定价，有安全边际 |
| 50x ~ 80x | 3分 | 估值合理，需增速匹配 |
| 80x ~ 150x | 2分 | 估值偏高 |
| > 150x | 1分 | 严重高估或AI占比过低 |

### 特殊情况处理
- 如果AI占比 < 10%，此指标会非常高，自动给1分
- 如果AI占比 > 80%，此指标接近PE，可交叉验证
- 如果净利润为负，跳过此维度，权重分配给其他维度

### 输出示例
```
市值/AI利润比评分: 2分
- 市值: 6022亿
- 净利润: 71.32亿
- AI收入占比: 85%（来源：2025年报）
- AI利润: 60.62亿
- 比值: 99.4x → 2分（估值偏高）
```

| AI收入占比 | 评分 | 示例公司 |
|-----------|------|---------|

## 维度6：护城河 (moat) - 权重20%

### 自动化程度: ❌ 0%自动化（纯LLM分析）

### 数据获取

**LLM搜索关键词：**
```
- "{公司名}市占率"

- "{公司名}竞争对手"
- "{公司名}核心技术专利"
- "{公司名}客户认证周期"
- "{行业}竞争格局"
```

### 评分规则（LLM判断）

| 护城河强度 | 评分 | 特征 |
|-----------|------|------|
| 全球垄断/寡头 | 5分 | 全球前2，技术代差明显，客户无法切换 |
| 国内龙头+全球前5 | 4分 | 有明显技术或规模壁垒，客户粘性强 |
| 行业前10 | 3分 | 有一定竞争力，但可被替代 |
| 有差异化 | 2分 | 有特色但壁垒不高 |
| 完全竞争 | 1分 | 同质化严重，价格战 |

### 护城河类型判断

**技术壁垒:**
- 专利数量、研发投入占比
- 良率优势、工艺领先
- 核心人才团队

**规模效应:**
- 产能规模、成本优势
- 市占率趋势（上升/下降）

**客户粘性:**
- 认证周期（>1年=强）
- 长期合作年限
- 切换成本高低

### 输出示例
```
护城河评分: 5分
- 全球市占率: 800G光模块35%（第一）
- 护城河类型: 技术壁垒 + 客户粘性
- 核心优势: 良率领先、产能最大、英伟达深度绑定
- 竞争格局: 中际旭创>新易盛>光迅科技
```

### 评分规则（LLM判断）

| 催化/风险平衡 | 评分 | 特征 |
|--------------|------|------|
| 重大催化+风险可控 | 5分 | 有明确的近期利好事件，无重大风险 |
| 有催化+小风险 | 4分 | 潜在催化剂存在，风险较小 |
| 平衡 | 3分 | 催化剂和风险都一般 |
| 催化不明+有风险 | 2分 | 缺乏催化剂，存在一定风险 |
| 重大风险 | 1分 | 有重大悬而未决的风险 |

### 催化剂类型（加分项）
- 新产品量产（如1.6T光模块）
- 新客户突破
- 产能释放/扩产落地
- 行业政策利好
- 业绩超预期
- 机构上调目标价

### 风险类型（减分项）
- 大客户依赖（单一客户>50%）
- 技术迭代风险（被替代）
- 地缘政治风险（制裁）
- 解禁/减持压力
- 行业产能过剩
- 应收账款/存货风险

### 输出示例
```
催化剂/风险评分: 4分
催化剂:
- 1.6T光模块2026年放量
- 高盛上调目标价至791元
- 产能持续扩张

风险:
- 英伟达依赖度高（约60%）
- 估值已较高
```

## 评估执行流程

### 设计原则：程序能做的程序做，减少LLM介入

```
✅ 正确做法：脚本直接输出人类可读格式 → 原样返回用户

```

### 单股评估标准流程

```
步骤1: 获取基础数据（脚本自动）
├── stock-data search {名称} → 获取代码
├── stock-data profile {代码} → 公司简介

步骤2: 计算自动化维度
├── 财务质量: ROE/毛利率/净利率/负债率 → 评分
└── 估值水平: PE/PEG → 评分

步骤3: LLM搜索分析
├── AI含量: 搜索AI收入占比、客户结构
├── 护城河: 搜索市占率、竞争格局
├── 需求增长: 搜索行业CAGR（公司增速已自动获取）
└── 催化剂/风险: 综合新闻、研报

步骤4: 计算半自动维度
└── 市值/AI利润比: 市值(自动) ÷ (利润(自动) × AI占比(搜索))

步骤5: 汇总评分
├── 计算加权总分
├── 确定星级
└── 生成评估报告

步骤6: 更新数据库
└── db.py update {代码} -d {维度} -s {分数} -e "依据"

步骤7: 等待用户评级
└── db.py rate {代码} -s {用户星级} -n "用户备注"
```

### 问题1: JSON数据库并发写入损坏

**症状:**
```
json.decoder.JSONDecodeError: Extra data: line X, column Y
```

**原因:**
多个并行的 `db.py update` 命令同时读写同一个JSON文件，导致文件内容被重复追加。

**解决方案:**
1. ✅ **已实现**: db.py 使用文件锁机制（Windows: msvcrt, Unix: fcntl）
2. ✅ **已实现**: 提供 `batch` 命令一次性更新多个维度
3. **操作建议**: 优先使用 `batch` 命令，避免并行执行多个 `update` 命令

**修复损坏的JSON:**
```python
# 手动检查文件末尾，删除重复的 }]}
# 常见损坏形式: "}\n  ]\n}  }\n  ]\n}" 应改为 "}\n  ]\n}"
```

### 问题3: Evidence字段特殊字符

**症状:**
evidence 字段包含特殊字符（如引号、百分号）时可能导致命令解析错误。

**解决方案:**

- 避免使用中文、引号、百分号等特殊字符
- 示例: `Rev33pct-Profit15pct` 而非 `营收增长33%，利润增长15%`

### 问题4: Windows控制台编码问题

**症状:**
```
UnicodeEncodeError: 'gbk' codec can't encode character '\u2b50' in position 0
```

**原因:**
Windows PowerShell 默认使用 GBK 编码，无法显示星星符号 `⭐` 等 Unicode 字符。

**解决方案 (已在脚本中实现):**
```python
import sys
import io
if sys.platform == 'win32':
    sys.stdout = io.TextIOWrapper(sys.stdout.buffer, encoding='utf-8', errors='replace')
```

**备选方案:**
- 在 PowerShell 中运行: `chcp 65001` 切换到 UTF-8 编码
- 使用 Windows Terminal（默认支持 UTF-8）
